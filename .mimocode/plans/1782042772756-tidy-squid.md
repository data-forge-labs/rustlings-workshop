## Key Differences: Article vs Existing Radish

| Aspect | Article's Redis Clone | Radish (existing) |
|--------|----------------------|-------------------|
| **Threading** | Multi-threaded Tokio (`Arc<Mutex<>>`) | Single-threaded Tokio (`Rc<RefCell<>>`, `spawn_local`) |
| **Protocol** | Simplified text protocol | Full RESP byte-level parsing |
| **Persistence** | JSON snapshots every 30s | None (deferred to exercises) |
| **CLI Client** | Built-in Rust client | Uses external `redis-cli` |
| **Error Handling** | `thiserror` custom errors | Minimal error focus |
| **Commands** | SET, GET, DEL, EXISTS, EXPIRE, TTL, DBSIZE, PING, QUIT | SET, GET, TTL, PING, ECHO |
| **Tests** | Unit + integration tests | Exercises only |
| **Background Tasks** | `tokio::spawn` + `time::interval` for expiry | None |
| **Dependencies** | tokio, serde, serde_json, thiserror | tokio, bytes, chrono |

## What the Article Uniquely Teaches

1. **Multi-threaded async TCP server** — Radish teaches `Rc<RefCell<>>` (single-threaded). This teaches `Arc<Mutex<>>` (multi-threaded), which is the more common production pattern.

2. **Disk persistence with serde** — Radish explicitly lists "Add persistence" as a next step. This delivers it with JSON snapshots.

3. **CLI client development** — Radish lists "Add a client" as a next step. This builds a complete Rust CLI client.

4. **Background tasks** — `tokio::spawn` with `time::interval` for periodic cleanup. Not covered elsewhere in the course.

5. **Comprehensive testing** — Unit tests for the parser + integration tests for the full command execution path. Radish has exercises but no built-in test suite.

6. **`thiserror` in production** — Radish doesn't focus on custom error types. This shows the `thiserror` derive macro pattern.

## Recommended Placement

**Section 10: ProductionSystems → Project 58: TinyRedis**

Place it BEFORE Radish in the learning path:

```
Project 58: TinyRedis (simpler, multi-threaded, Arc<Mutex<>>)
    ↓ builds on
Project 59: Radish (advanced, single-threaded, Rc<RefCell<>>, RESP)
```

### Why before Radish?

- TinyRedis is simpler (no RESP protocol, no `bytes` crate)
- Teaches `Arc<Mutex<>>` which students already learned in Section 5
- Radish then shows the advanced single-threaded approach with `Rc<RefCell<>>`
- Students see two valid architectures for the same problem
